# Traversal Contract — `traversal_v1`

> Defines the deterministic ordering rules that all retrieval operations must follow.
> Any change to these rules requires incrementing `TRAVERSAL_CONTRACT_VERSION`.

## Core ordering rules

### Arrays

Traversed in **ascending index order**: `[0], [1], [2], ...`

Implementation: `enumerate(value)` yields `(index, element)` pairs in natural order.

### Objects

Traversed in **lexicographic key order**: keys sorted via Python `sorted()`.

Implementation: `sorted(value)` yields keys in Unicode code-point order.

### Recursive traversal

Depth-first. At each level:
1. If array → ascending index
2. If object → lexicographic keys
3. Scalars are leaf nodes — no further traversal

## Path construction

Paths are built from `$` (root) using these rules:

| Key type | Format | Example |
|----------|--------|---------|
| ASCII identifier (`[A-Za-z_][A-Za-z0-9_]*`) | Dot notation | `$.data.name` |
| Non-identifier key | Bracket notation | `$['special-key']` |
| Array index | Bracket notation | `$[0]`, `$[1]` |
| Mixed | Combined | `$.data[0].field['non-id']` |

### Escape rules for bracket notation

| Sequence | Meaning |
|----------|---------|
| `\\` | Literal backslash |
| `\'` | Literal single quote |
| `\n` | Newline |
| `\r` | Carriage return |
| `\t` | Tab |

## Wildcard `[*]` expansion

Wildcards expand following the same ordering rules as their container:

- **Array `[*]`**: expands in ascending index order
- **Object `[*]`**: expands in lexicographic key order

A total expansion cap (`max_wildcard_expansion_total`) limits the combined number of nodes generated by all wildcards in a single query. Exceeding the cap raises `JsonPathError`.

## JSONPath subset

Only the following grammar is supported:

| Segment | Syntax | Example |
|---------|--------|---------|
| Root | `$` | `$` |
| Field (dot) | `.name` | `$.data` |
| Field (bracket) | `['name']` | `$['special-key']` |
| Index | `[n]` | `$[0]` |
| Wildcard | `[*]` | `$.items[*]` |

### Caps

| Parameter | Purpose |
|-----------|---------|
| `max_jsonpath_length` | Max length of entire JSONPath string |
| `max_path_segments` | Max number of segments (depth) |
| `max_wildcard_expansion_total` | Max total wildcard expansion results |

## Select paths normalization

When `select_paths` are provided to retrieval tools:

1. Each path is canonicalized via `canonicalize_jsonpath()`
2. Paths are deduplicated (set operation)
3. Paths are sorted lexicographically
4. `select_paths_hash = sha256(canonical_bytes(sorted_paths))` (RFC 8785)

This ensures identical sets of paths always produce the same hash regardless of input order.

## Sampled mode enumeration (partial mapping)

When operating on partially mapped artifacts:

1. `sample_indices` are re-sorted to ascending order regardless of storage order
2. Indices outside valid range (`0 <= idx < len(records)`) are silently skipped
3. Enumeration yields `(path, sample_index, record)` tuples
4. Sampled records are NOT recursively expanded — each is a leaf

## Determinism guarantees

Given identical:
- Artifact payload (`payload_hash_full`)
- Query parameters (paths, where clause, pagination position)
- Version constants (`traversal_contract_version`, `mapper_version`, `map_budget_fingerprint`)

The traversal will produce:
- Identical item boundaries
- Identical item ordering
- Identical cursor positions
- Identical `select_paths_hash` values

Any version constant change invalidates all existing cursors bound to the old version.
